## 1. 产品概述

### 1.1 一句话描述
微信小程序，将文学考研教材中的古诗文名句做成填空式闪卡，按艾宾浩斯记忆曲线科学排期复习。

### 1.2 产品愿景
解决文学考研备考中"古诗文背诵量大、遗忘快、缺乏科学排期"的核心痛点。把复习节奏交给算法，考生只需每天打开小程序完成当日任务即可。

### 1.3 使用场景
- 考研党每日碎片时间（排队、通勤、睡前）
- 冲刺阶段高强度复习
- 教材学完一个模块后的针对性巩固

## 2. 目标用户

| 维度 | 描述 |
|------|------|
| 核心用户 | 文学考研考生（文艺学、古代文学、现当代文学等方向） |
| 年龄 | 21-25岁，二战/三战比例偏高 |
| 设备 | 主力用手机，微信重度使用 |
| 痛点 | 背诵内容多（300+首诗词）、遗忘快、缺乏系统复习工具 |
| 现有方案 | 手抄卡片、Anki（上手成本高）、打印资料（无法交互） |

## 3. 产品定位与边界

### 3.1 定位
**纯个人记忆工具**，不是内容社区，不是知识付费平台。

### 3.2 明确不做
- ❌ 社区/UGC公开分享
- ❌ 微信支付 / 虚拟商品购买
- ❌ 公共内容池（版权风险）
- ❌ AI自动生成题目（不靠谱）
- ❌ 社交裂变（打卡分享朋友圈等）

### 3.3 合规
- 个人主体小程序，不涉及支付和虚拟商品
- 古诗原文（公有领域）自由使用
- 教材论述需提炼概括，标注出处，不可复制大段原文

### 4.2 卡片三层结构

每张卡片包含三层信息，复习时逐层展开：

```
第一层：填空句
┌─────────────────────────────┐
│  "____________，春风吹又生。" │
│  [选项A] [选项B] [选项C] [选项D] │
└─────────────────────────────┘

第二层：原文全文
┌─────────────────────────────┐
│  离离原上草，一岁一枯荣。     │
│  野火烧不尽，春风吹又生。     │
│  远芳侵古道，晴翠接荒城。     │
│  又送王孙去，萋萋满别情。     │
└─────────────────────────────┘

第三层：论述题用法 + 出处标注
┌─────────────────────────────┐
│  出处：白居易《赋得古原草送别》 │
│  朝代：唐                     │
│  用法：可用于"生命意识""韧性"  │
│  等论述题，与李商隐"春蚕到死    │
│  丝方尽"形成对比。             │
│  参考：袁行霈《中国文学史》     │
│  第二卷 Pxxx                  │
└─────────────────────────────┘
```

## 5. 三种记忆考察模式

三种模式**平级选择**，非递进关系。每次开始复习时由用户选择本次使用哪种模式。

### 5.1 模式一：选词填空
- **适用场景**：刚学新内容，还不太熟
- **机制**：给出挖空句子，4选1选择正确答案
- **认知层级**：再认记忆（recognition）
- **交互**：点击选项即时反馈对错

### 5.2 模式二：遮挡复现（Anki式翻转）
- **适用场景**：有一定印象但不够牢固
- **机制**：先显示挖空句子 → 用户脑中回想 → 点一下显示答案 → **自评**
- **自评环节**（必须有，否则无效）：
  - 按钮A：「我想对了」→ 进入下一阶段
  - 按钮B：「没想起来」→ 回退到阶段0
- **认知层级**：回忆记忆（recall）

### 5.3 模式三：整句默写
- **适用场景**：需要精准掌握原文
- **机制**：给出上下文提示 → 全句打字输入 → 不做自动判分
- **判分方式**：用户自行比对标准答案
- **设计理由**：AI判分无法准确判断"意思对但字不对"的情况，交由用户自判更可靠

### 5.4 模式选择交互
```
┌────────────────────────────┐
│  今日复习 · 12张卡片        │
│                            │
│  选择复习模式：             │
│  ○ 选词填空（4选1）        │
│  ○ 遮挡复现（自己回想）     │
│  ○ 整句默写（打字输入）     │
│                            │
│  [开始复习]                │
└────────────────────────────┘
```
模式选择放在**每次复习开始时**，不做全局设置（用户可能不同状态下想用不同模式）。

## 6. 卡片数据结构

### 6.1 Card对象

```javascript
{
  _id: String,              // 唯一标识
  // 正面（填空题面）
  front: {
    before: String,         // 挖空前的文本，如 ""
    blank: String,          // 正确答案，如 "野火烧不尽"
    after: String,          // 挖空后的文本，如 "，春风吹又生。"
  },
  
  // 答案
  answer: String,           // 正确答案（用于选词填空的选项比对）
  
  // 完整原文
  fullText: String,         // 包含上下文的完整诗句/段落
  
  // 元信息
  title: String,            // 篇名，如《赋得古原草送别》
  author: String,           // 作者，如 白居易
  dynasty: String,          // 朝代，如 唐
  
  // 论述题用法（提炼概括，非教材原文）
  essayQuestions: [{
    topic: String,          // 考点话题，如 "生命意识"
    usage: String,          // 用法说明（用户自己的话概括）
    relatedCards: [String], // 可关联的其他卡片ID
  }],
  
  // 出处标注
  sources: [{
    book: String,           // 书名，如 "袁行霈《中国文学史》第二卷"
    chapter: String,        // 章节
    note: String,           // 备注
  }],
  
  // 标签（多对多）
  tags: [String],           // 如 ["唐", "送别诗", "白居易", "高频考点"]
  
  // 卡片类型
  type: String,             // "fill_verse" | "fill_author" | "keyword_qa" | "full_recite"
  
  // 艾宾浩斯状态
  stage: Number,            // 当前阶段 0-7，0=新卡片/重置
  nextReview: Date,         // 下次复习日期
  reviewHistory: [{         // 复习历史
    date: Date,
    mode: String,           // "choice" | "flip" | "dictation"
    result: String,         // "correct" | "wrong"
  }],
  
  // 难度
  difficulty: Number,       // 1-5，用户可标记
  
  // 时间戳
  createdAt: Date,
  updatedAt: Date,
}
```

### 6.2 四种卡片类型

| 类型 | type值 | 说明 | 示例 |
|------|--------|------|------|
| 名句填空 | `fill_verse` | 挖掉诗句中的关键句 | "______，春风吹又生" |
| 作者/篇名填空 | `fill_author` | 给出诗句，填作者或篇名 | "此句出自______的《______》" |
| 简答关键词 | `keyword_qa` | 考点关键词触发记忆 | Q: "白居易新乐府运动核心主张" → A: 关键词列表 |
| 全文背诵 | `full_recite` | 篇名触发全文回忆 | 篇名 → 脑中默背全文 |

## 7. 艾宾浩斯记忆算法

### 7.1 七阶段排期

| 阶段 | 间隔 | 说明 |
|------|------|------|
| Stage 0 | 当天 | 新卡片，当天复习 |
| Stage 1 | 1天后 | 第一次成功 → 1天后 |
| Stage 2 | 2天后 | 第二次成功 → 2天后 |
| Stage 3 | 4天后 | 第三次成功 → 4天后 |
| Stage 4 | 7天后 | 第四次成功 → 7天后 |
| Stage 5 | 15天后 | 第五次成功 → 15天后 |
| Stage 6 | 30天后 | 第六次成功 → 30天后 |
| Stage 7 | 60天后 | 第七次成功 → 60天后（长期记忆） |

### 7.2 核心规则

1. **答对** → 阶段+1，按新间隔计算下次复习日期
2. **答错** → **回退到Stage 0**，从当天重新开始
3. **每日上限**：有每日最大复习量设置，防止积压过多
4. **新卡优先**：当日既有新卡又有旧卡复习时，新卡优先出现
5. **断档恢复**：如果中断多天，不会把积压全部堆给用户，而是：
   - 显示积压总数
   - 每日最多额外+5张（可配置）
   - 提供「快速热身」模式快速过一遍积压卡片

### 7.3 每日复习量计算

```
每日复习量 = 新卡(当日到期) + 复习卡(各阶段到期) 
上限 = 用户设定的每日上限（默认30张）
超限 → 复习卡优先，新卡顺延到次日
```

## 8. 标签体系

### 8.1 标签维度

| 维度 | 标签示例 | 用途 |
|------|---------|------|
| 朝代 | 先秦、两汉、魏晋、唐、宋、元、明、清 | 按文学史时期筛选 |
| 体裁 | 诗、词、曲、赋、散文 | 按文体筛选 |
| 题材 | 咏史、怀古、送别、山水田园、边塞、爱情、哲理 | 按主题筛选 |
| 作者 | 李白、杜甫、苏轼等 | 按作者筛选 |
| 考点标签 | 高频考点、真题出现过、名词解释常考 | 按考试重要性筛选 |
| 来源 | 袁版文学史、真题、自编 | 卡片数据来源 |

### 8.2 标签使用
- 一张卡片可以有多个标签
- 标签支持组合筛选（如「唐 + 送别诗 + 高频考点」）
- 标签由用户在导入时或编辑时手动添加，不做自动分类

## 10. 技术方案

### 10.1 总体架构

```
┌─────────────────────────────────┐
│       微信小程序前端              │
│   (原生框架 + WeUI组件库)         │
├─────────────────────────────────┤
│        微信云开发 CloudBase       │
│  ┌──────────┬──────────┬──────┐ │
│  │ 云数据库  │ 云函数    │ 云存储│ │
│  │ (卡片库) │ (排期计算)│(文件) │ │
│  └──────────┴──────────┴──────┘ │
└─────────────────────────────────┘
```

### 10.2 技术选型

| 层 | 选型 | 理由 |
|----|------|------|
| 前端框架 | 微信小程序原生 | 个人开发，原生足够，避免框架额外学习成本 |
| UI组件 | WeUI | 微信官方组件库，风格统一 |
| 后端/数据库 | 微信云开发 CloudBase | 免服务器运维，免费额度够用 |
| 文件存储 | 云存储 | .txt/.md文件存储 |
| 定时任务 | 云函数定时触发器 | 每日凌晨计算当日复习任务 |

### 10.3 免费额度（CloudBase个人版）

| 资源 | 免费额度 | 是否够用 |
|------|---------|----------|
| 云数据库容量 | 2GB | ✅ 足够 |
| 云函数调用次数 | 20万次/月 | ✅ 足够 |
| 云存储容量 | 5GB | ✅ 足够 |
| CDN流量 | 5GB/月 | ✅ 足够 |

| 内容类型 | 版权状态 | 使用方式 |
|----------|---------|----------|
| 古诗原文 | **公有领域**（作者去世超过50年） | 自由使用 |
| 教材论述 | **受版权保护** | 不可复制大段原文，只能提炼概括 + 标注出处（书名+章节+页码） |
| 用户自建内容 | **用户所有** | 仅存储在用户自己的云开发环境中 |
| 预置示例卡片 | **自行编写** | 选取公有领域古诗，论述部分自行概括 |

**规避风险的操作原则**：
- 教材论述字段只存概括性要点，不存原文段落
- 出处标注格式：「参见：袁行霈《中国文学史》第二卷 Pxxx」
- 不做公共内容池，每个用户的卡片数据隔离
